第 1 套：初级（SpringBoot 入门 + HTTP 协议 + Tomcat）
一、业务场景
开发一个“图书服务入门接口”，用于完成 SpringBoot 项目的快速搭建，并提供最基础的 HTTP 查询和新增接口。
二、题目要求
1. 创建一个 SpringBoot Web 项目；
2. 在配置文件中完成以下配置：
  - 服务端口改为 8081
  - 访问路径前缀设置为 /api
3. 定义 BookController，完成以下接口：
  - GET /api/books/hello：返回字符串 "springboot ok"
  - GET /api/books/{id}：根据 id 返回一本图书信息
  - POST /api/books：接收图书名称、作者、价格，返回新增成功的信息
4. 要求接口返回结果能够通过 Apipost / Postman 正常访问；
5. 图书数据可先使用假数据模拟，不要求接数据库。
三、验收标准
1. SpringBoot 项目能够正常启动
- 验收方式：
  - 项目启动无报错；
  - 控制台能看到内嵌 Tomcat 启动成功；
  - 浏览器或接口测试工具可正常访问服务。
- 设计理由：
SpringBoot 的第一步不是业务逻辑，而是先把项目跑起来。只有启动链路、依赖、端口、Tomcat 容器都没问题，后面的接口开发才有基础。
2. HTTP 接口定义正确
- 验收方式：
  - GET 接口能完成查询；
  - POST 接口能完成新增；
  - 请求方式与接口语义保持一致。
- 设计理由：
接口开发不只是把路径写出来，更重要的是请求方式和业务动作匹配。查询、提交如果混着写，后面接口规范会越来越乱。
3. 路径与配置生效
- 验收方式：
  - 服务端口确实变为 8081；
  - 访问路径前缀确实为 /api；
  - 接口实际访问地址与配置一致。
- 设计理由：
配置文件是否真正生效，是 SpringBoot 开发里的基础能力。很多问题表面看像代码错，实际是配置没起作用。
4. 返回结果清晰可读
- 验收方式：
  - 字符串接口直接返回明确内容；
  - 查询图书接口返回结构清晰的 JSON；
  - 不出现空响应、乱码、格式混乱等问题。
- 设计理由：
接口最终是给前端或测试工具使用的，结果是否清晰会直接影响联调效率。哪怕是入门题，也要养成良好的输出习惯。
